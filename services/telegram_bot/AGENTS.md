# СЕРВИС: TELEGRAM BOT (NOTIFIER & INTERACTION)

Интерактивная фронтенд-панель управления откликами на базе `aiogram 3`. Сервис работает как потребитель (consumer) задач из `arq`, отправляет карточки вакансий пользователю и обрабатывает двухэтапное логирование решений в PostgreSQL.

## 1. Технический стек и зависимости
- **Фреймворк:** `aiogram 3.x` (строго асинхронный).
- **Менеджер задач:** `arq` (слушает выделенную очередь `telegram_queue`).
- **База данных:** `SQLAlchemy 2.0` (асинхронные сессии через `shared`).
- **Форматирование:** Строго HTML-разметка сообщений.

## 2. Архитектурные правила и паттерны

### 2.1. Безмашинная навигация (Stateless Callbacks)
- **Запрещено:** Использовать FSM (Finite State Machine) для фиксации этапов разбора вакансий. Бот должен быть устойчив к перезапускам.
- **Разрешено:** Передавать ID вакансии и текущий шаг (`action`) внутри `CallbackData` (максимум 64 байта на строку колбэка согласно лимитам Telegram API).

### 2.2. Защита данных и UX
- **Двухэтапное подтверждение:** Любое действие пользователя (Отклик/Отказ) требует промежуточного шага подтверждения (`Подтвердить ...` / `Назад`). Статус в БД меняется только после явного подтверждения.
- **Сбор метрик (Обратная связь):** После фиксации статуса клавиатура ОБЯЗАНА переключиться на выбор причины. Текст выбранной причины пишется в текстовое поле `notes` модели `Vacancy`.
- **Копирование откликов:** Текст сопроводительного письма (`cover_letter`) должен быть обязательно обернут в тег `<code>...</code>` для копирования на мобильных устройствах в один тап.

## 3. Интеграция с очередью (arq)
- Сервис регистрирует таску `send_vacancy_notification(ctx, vacancy_data: dict)`.
- Объект `Bot` инициализируется один раз при старте воркера и пробрасывается в контекст `ctx["bot"]`.
- Вызов таски происходит из сервиса `analyzer` сразу после успешного ИИ-анализа текста и генерации сопроводительного письма.
