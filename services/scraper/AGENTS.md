# ИНСТРУКЦИЯ ДЛЯ СЕРВИСА SCRAPER (Network I/O)

## 1. Сетевой слой и Лимиты
- Для всех запросов использовать `curl_cffi.requests.AsyncSession`.
- **Лимит:** Строго не более 5 запросов за 6 секунд. Запросы должны быть равномерно размыты по таймлайну. Перед каждым вызовом делать:
  `await asyncio.sleep(random.uniform(1.0, 1.5))` (базовый интервал ~1.2с + джиттер).

## 2. Динамическая Мультиплатформенность (Селекторы)
- Запрещено хардкодить CSS-селекторы и логику под конкретный сайт.
- При старте воркер загружает конфигурацию из `/app/selectors.json`.
- Реализовать динамический резолв платформы: код должен анализировать входящий URL через `urllib.parse.urlparse`, сопоставлять его с ключами верхнего уровня в `selectors.json` (например, `hh.ru`, `career.habr.com`) и использовать для парсинга только соответствующую ветку конфигурации (`searcher` или `parser`).
- Если домен не найден в файле конфигурации, генерировать информативное исключение и переводить ссылку в статус `failed`.

## 3. Логика работы и стандартизация данных
- При извлечении платформы из URL приводить её к единому системному идентификатору (slug): `hh` для hh.ru, `habr` for career.habr.com.
- `search.py`: Заполняет `vacancy_links` (поля `url`, `platform`, `status='new'`).
- `pipeline.py`: При создании записи в `vacancies` обязательно пробрасывает соответствующее значение в поле `platform`.
